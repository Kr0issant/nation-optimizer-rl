# Proposed Amendment — Training Specification Alignment with Game Mechanics

> Status: **Under deliberation**  
> Scope: `specification/13_RL_ADAPTERS_AND_TRAINING.md`, `training/reward_fn.py`, `training/train_grpo.py`, `scripts/collect_grpo_prompts.py`, `llm_integration/prompts/minister.py`  
> Impact: HIGH — affects the entire training pipeline and reward signal  
> Dependencies: Resolves AFTER `01-critical-threshold-abstention-voting-fix.md` and `02-readme-veracity-and-theme-alignment.md` are merged  

---

## 1. Problem Statement

The current training pipeline (on the `train` branch) assumes the **old game mechanics** where:
- Ministers propose **total allocations** (not discretionary)
- `ABSTAIN_FROM_PROPOSAL` is a valid action
- Critical failure triggers at `Allocation < Critical`
- Rejection = allocation = 0 = death

If we adopt **Option A** from `01-critical-threshold-abstention-voting-fix.md` (auto-fund critical + discretionary proposals), the training pipeline will train against **stale mechanics**. The model will learn to play a game that no longer exists.

**This amendment documents all downstream training changes needed per game-mechanic option.** It should be merged to `main` as a reference document so that training PRs can be opened after the core mechanic amendments are resolved.

---

## 2. Current Training Pipeline (as of `train` branch)

### Algorithm
- **GRPO** (Group Relative Policy Optimization) via `trl.GRPOTrainer`
- Base model: `Qwen/Qwen2.5-0.5B-Instruct`
- LoRA: `r=16, alpha=32, dropout=0.05` on `q_proj, k_proj, v_proj, o_proj`
- Training script: `training/train_grpo.py`

### Dataset
- Generated by `scripts/collect_grpo_prompts.py`
- Source: Rollouts from `OptimalZoneAdapter` baseline
- Phases included: `PROPOSAL` (Phase 3) and `VOTING` (Phase 4)
- Phase 2 (`DEBATE`) is **inference-only** — no debate prompts enter the training dataset
- Format: JSONL with `prompt`, `agent_id`, `valid_actions`, `treasury`, `own_sector` (critical/demand/surplus), `proposals`, `all_sectors`

### Reward Function (`training/reward_fn.py`)

| Action | Reward |
|--------|--------|
| Parse fail | -1.0 |
| Illegal action | -0.5 |
| Abstain proposal | 0.0 |
| Propose budget | `0.2 + 0.9 × (RF - 1.0)` |
| Vote YES | `RF - 1.0` |
| Vote NO | `-(RF - 1.0)` |
| Vote ABSTAIN | 0.0 |

**Key property**: Reward is grounded in `core.revenue.revenue_factor()` using the same piecewise math as the simulator. No reward hacking possible.

### Prompt (`llm_integration/prompts/minister.py`)
- Role: "Minister for [department]"
- Instructions: "EVERY department MUST submit a proposal — you are not allowed to abstain"
- Debate history: last 10 messages included
- Action schemas: `DEBATE`, `PROPOSE_BUDGET`, `VOTE`, `FINISH_DEBATE`
- Output: Strict JSON, no prose outside JSON

## 3. Downstream Changes per Game-Mechanic Option

### Option A: Auto-Fund Critical + Discretionary Proposals (RECOMMENDED)

If Option A is adopted, the following training changes are required:

#### 3A.1 Remove `ABSTAIN_FROM_PROPOSAL` from training dataset and reward

**Files:**
- `schemas/actions.py` — remove `ABSTAIN_FROM_PROPOSAL` enum value and `AbstainProposalAction` dataclass
- `schemas/phases.py` — change `Phase.PROPOSAL` valid actions to `frozenset({"PROPOSE_BUDGET"})`
- `training/reward_fn.py` — remove `ABSTAIN_REWARD` branch; illegal actions should remain at `-0.5`, but there is no longer a neutral "abstain" path
- `scripts/collect_grpo_prompts.py` — ensure `OptimalZoneAdapter` never emits abstain actions
- `agents/rule_based/` — update all baselines to never abstain (or raise error if no valid action)

**Rationale:** Under Option A, every minister must submit a proposal every round. Abstention is a non-action.

#### 3A.2 Update reward function to score discretionary proposals

**Current scoring (PROPOSE_BUDGET):**
```python
rf = _piecewise_revenue_factor_for_amount(action.amount, own_sector)
return PROPOSE_BASE_REWARD + PROPOSE_FACTOR_WEIGHT * (rf - 1.0)
```

**New scoring (PROPOSE_BUDGET under Option A):**
```python
# Total allocation = auto_critical + discretionary
total_allocation = own_sector["critical"] + action.amount
rf = _piecewise_revenue_factor_for_amount(total_allocation, own_sector)
return PROPOSE_BASE_REWARD + PROPOSE_FACTOR_WEIGHT * (rf - 1.0)
```

**Key changes:**
- `action.amount` is now **discretionary** (additional funding above critical)
- The revenue factor is computed on **total allocation**, not just the proposed amount
- Treasury constraint changes: `0.0 <= action.amount <= (treasury - total_critical)` because critical is auto-deducted before discretionary

**File:** `training/reward_fn.py`

#### 3A.3 Update treasury constraint in reward function

**Current:**
```python
treasury = _safe_float(row.get("treasury"), default=0.0)
if not 0.0 <= float(action.amount) <= treasury:
    return ILLEGAL_ACTION_REWARD
```

**New:**
```python
treasury = _safe_float(row.get("treasury"), default=0.0)
total_critical = sum(sector["critical"] for sector in all_sectors.values())
if not 0.0 <= float(action.amount) <= (treasury - total_critical):
    return ILLEGAL_ACTION_REWARD
```

**Rationale:** Under Option A, the treasury must reserve `total_critical` for auto-funding. Discretionary proposals compete for the remainder.

**File:** `training/reward_fn.py`

#### 3A.4 Update vote reward semantics

**Current vote scoring:**
- Vote YES on proposal with RF > 1.0: positive reward
- Vote NO on proposal with RF > 1.0: negative reward (because rejection = death)

**New vote scoring (Option A):**
- Vote YES on proposal with RF > 1.0: positive reward (approves discretionary growth)
- Vote NO on proposal with RF > 1.0: small negative reward (rejection = survival only, not death)
- The magnitude of the NO penalty can be **reduced** because rejection is no longer catastrophic

**Suggested new formula:**
```python
if action.vote is VoteChoice.YES:
    return VOTE_YES_WEIGHT * (rf - 1.0)  # default VOTE_YES_WEIGHT = 1.0
if action.vote is VoteChoice.NO:
    return VOTE_NO_WEIGHT * -(rf - 1.0)   # default VOTE_NO_WEIGHT = 0.5 (less severe)
```

**Rationale:** Voting NO under Option A is a strategic choice (fund growth vs. conserve treasury), not a murder weapon. The penalty should reflect reduced stakes.

**File:** `training/reward_fn.py`

#### 3A.5 Add episode-level reward bonus (optional but recommended)

**Current**: GRPO uses only per-action dense rewards. There is no signal that early-round decisions affect late-round outcomes.

**Problem**: A model can learn to make good individual proposals but never learn to buffer treasury for black swan events.

**Suggested addition** (requires dataset to include episode termination metadata):
```python
# At episode end, add a terminal bonus to all actions in the episode
terminal_bonus = SURVIVAL_BONUS_WEIGHT * (rounds_survived / max_rounds)
# Apply as a constant offset to all per-step rewards in the episode
```

**Implementation options:**
1. **Two-stage training**: Stage 1 = per-action GRPO (current). Stage 2 = episode-level PPO fine-tuning.
2. **Hybrid reward**: GRPO reward function reads `rounds_survived` from dataset metadata and adds a small terminal bonus to every action in the episode.

**File:** `training/reward_fn.py` or a new `training/episode_reward.py`

#### 3A.6 Update minister prompt to explain auto-critical

**Current prompt excerpt:**
> "PROPOSE your own departmental needs based on the events in the 'Observation State'. EVERY department MUST submit a proposal — you are not allowed to abstain."

**New prompt excerpt:**
> "Your department automatically receives its **critical minimum** funding (40% of demand) each round. You are now proposing **additional discretionary funding** above this baseline. Propose an amount ≥ 0. A proposal of 0 means you need only the critical minimum this round. The treasury has already reserved critical funding for all departments. Your discretionary proposal competes for the remaining treasury."

**Also update:**
- Remove references to "abstain" or "skip your turn"
- Clarify that voting NO on a proposal means the department gets only critical funding (not death)

**File:** `llm_integration/prompts/minister.py`

#### 3A.7 Regenerate training dataset

**Current dataset**: `assets/datasets/grpo_prompts.jsonl` generated from `OptimalZoneAdapter`

**Required changes:**
1. Update `OptimalZoneAdapter` to emit discretionary amounts (not total allocations)
2. Re-run `scripts/collect_grpo_prompts.py`
3. Push new dataset to Hugging Face Hub (new version tag)
4. Update `DEFAULT_DATASET_ID` in `training/train_grpo.py`

**Files:**
- `agents/rule_based/optimal_zone.py`
- `scripts/collect_grpo_prompts.py`
- `training/train_grpo.py`

#### 3A.8 Update benchmark baselines

The benchmark scripts in `evaluation/benchmark_policies.py` must reflect the new mechanics:
- Remove abstention-based tests
- Adjust expected survival curves (auto-critical means no sudden death from abstention)
- Add new metric: "discretionary approval rate" (how often parliament approves growth funding)

**File:** `evaluation/benchmark_policies.py`

### Option B: Safe Abstention + Restricted Critical Check (Match Current Code)

If Option B is adopted (keep current code behavior, update spec to match), **no training changes are needed.** The current training pipeline already assumes:
- Abstention is safe (0 allocation, no critical check)
- Critical failure only applies to approved departments
- Voting NO is safe for the target department

**Verdict:** Training pipeline is already compatible. Only spec updates needed.

---

### Option C: Default Critical on Rejection/Abstention

If Option C is adopted (rejected/abstained departments get critical auto-funded):

#### 3C.1 Reward function changes
- Proposal scoring stays mostly the same (total allocation = proposed amount)
- But rejection no longer means allocation = 0 → the model doesn't directly control whether critical is received
- Vote scoring becomes harder to calibrate: voting NO means "reduce to critical", not "kill department"
- Suggested: reduce NO penalty magnitude (same as Option A)

#### 3C.2 Prompt changes
- Update prompt: "If your proposal is rejected, you will still receive critical minimum funding."
- Keep abstention as valid action (since it now means "accept critical only")

**Complexity:** Moderate. The "rejected but funded" mechanic is confusing for both the model and the trainer.

---

### Option D: Remove Critical Failure, Replace with Gradual Penalty

If Option D is adopted:

#### 3D.1 Remove critical penalty from reward
- `Critical_Penalty = -1000` is deleted
- Episode no longer terminates on critical underfunding
- Reward function focuses on prosperity + productivity + survival bonus only

#### 3D.2 Add productivity penalty to reward
- Below critical: apply `-10 × (1.0 - RF)` per sector (since RF is undefined below critical, use a proxy)
- Or use the environment's productivity drift as the implicit penalty

#### 3D.3 Dataset changes
- Episodes no longer end suddenly → longer episodes, more data per rollout
- Regenerate with longer max rounds

**Complexity:** Low to moderate. Simpler reward landscape but may make learning too easy.

---

### Option E: Mandatory Proposals + Consensus-Only Voting

If Option E is adopted:

#### 3E.1 Remove voting from training entirely
- Voting becomes ceremonial → no training signal
- Remove Phase 4 (`VOTING`) from dataset collection
- Train only on Phase 3 (`PROPOSAL`) and Phase 2 (`DEBATE`)

#### 3E.2 Add debate to training (major change)
- Currently debate is inference-only
- Would need to define a reward signal for debate messages (e.g., coherence, relevance, persuasiveness)
- This is a research problem in itself — NLP reward models are hard

**Complexity:** Very high. Not recommended for hackathon timeline.

---

### Option F: Minimum Viable Treasury Guarantee

If Option F is adopted:

#### 3F.1 Simplify reward function
- Treasury guarantee removes bankruptcy risk → reward becomes purely prosperity maximization
- Remove treasury constraint checks from reward
- Model can propose arbitrarily high amounts without fear

**Complexity:** Low. But the game becomes trivial and uninteresting for training.

## 4. Recommended Change Order

If **Option A** is chosen (recommended), apply changes in this order:

1. **Game engine changes first** (before any training):
   - `core/game.py`: auto-fund critical, remove abstention handler
   - `core/reward.py`: remove critical failure termination
   - `schemas/actions.py`, `schemas/phases.py`: remove abstention
   - `agents/`: update all rule-based adapters
   - Tests: update integration tests

2. **Training infrastructure changes second**:
   - `training/reward_fn.py`: update scoring formulas
   - `llm_integration/prompts/minister.py`: update prompt text
   - `scripts/collect_grpo_prompts.py`: update dataset generation
   - `agents/rule_based/optimal_zone.py`: emit discretionary amounts

3. **Dataset regeneration third**:
   - Re-run `collect_grpo_prompts.py`
   - Push new dataset to HF Hub
   - Update `train_grpo.py` dataset ID

4. **Re-training fourth**:
   - Run `train_grpo.py` from scratch (or resume from checkpoint if compatible)
   - Compare new reward curves against old baseline

5. **Benchmarking fifth**:
   - Update `evaluation/benchmark_policies.py`
   - Generate new baseline plots

## 6. Open Questions for Deliberation

1. **Should the debate phase ever enter training?**
   - Currently: No (inference only)
   - Option: Add debate reward using an external NLP model (e.g., coherence scorer)
   - Risk: Adds complexity, may not improve planning quality

2. **Should we keep the current GRPO reward range `[-1.0, +1.4]`?**
   - Under Option A, discretionary proposals could yield higher RF → potentially higher max reward
   - May need to recalibrate `PROPOSE_BASE_REWARD` and `PROPOSE_FACTOR_WEIGHT`

3. **Should terminal episode bonus be added to GRPO or kept for a second PPO stage?**
   - GRPO with terminal bonus: simpler, single-stage training
   - Two-stage (GRPO → PPO): better credit assignment but more complex

4. **How should the new dataset be versioned on HF Hub?**
   - Option: `nation-parliamentary-prompts-v2` (new dataset)
   - Option: Same dataset ID with new revision tag

5. **Should we train a separate model for Dictator vs. Parliamentary modes?**
   - Currently: One training pipeline for Parliamentary mode only
   - Option: Train Dictator separately with direct-allocation prompts
   - Benefit: Enables true comparison between reasoning structures
